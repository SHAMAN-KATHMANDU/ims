# Database Migration Patterns

Safe, zero-downtime database migration strategies for production PostgreSQL databases.

## When to Activate

- Adding new columns or tables
- Renaming columns or tables
- Changing column types
- Adding or removing indexes
- Migrating data between schemas
- Planning schema changes for production

## Core Principle: Expand-Contract

Never make breaking schema changes in a single deployment. Use the expand-contract pattern:

```
Phase 1 (Expand):   Add new column/table (backward compatible)
Phase 2 (Migrate):  Backfill data, update application code
Phase 3 (Contract): Remove old column/table (after all code uses new)
```

## Prisma Migration Safety

### Safe Operations (no downtime)

```sql
-- Adding a nullable column
ALTER TABLE users ADD COLUMN phone VARCHAR(20);

-- Adding a column with a default
ALTER TABLE users ADD COLUMN is_active BOOLEAN NOT NULL DEFAULT true;

-- Adding a new table
CREATE TABLE user_preferences (...);

-- Adding an index (use CONCURRENTLY in production)
CREATE INDEX CONCURRENTLY idx_users_email ON users(email);
```

### Dangerous Operations (require expand-contract)

```sql
-- Renaming a column (breaks existing queries)
ALTER TABLE users RENAME COLUMN email TO email_address;

-- Dropping a column (irreversible data loss)
ALTER TABLE users DROP COLUMN phone;

-- Changing column type (may fail or cause data loss)
ALTER TABLE users ALTER COLUMN age TYPE BIGINT;

-- Adding NOT NULL to existing column (fails if nulls exist)
ALTER TABLE users ALTER COLUMN phone SET NOT NULL;
```

## Expand-Contract Examples

### Renaming a Column

```
# Phase 1 — Expand: Add new column, keep old
prisma migrate dev --name add_email_address_column

# schema.prisma
model User {
  id           String @id
  email        String  # old — keep for now
  emailAddress String? # new — nullable during transition
}
```

```typescript
// Phase 2 — Application: Write to both, read from new
async updateUser(id: string, data: UpdateUserDto) {
  return prisma.user.update({
    where: { id },
    data: {
      email: data.email,        // write to old
      emailAddress: data.email, // write to new
    },
  });
}

// Read from new with fallback to old
const email = user.emailAddress ?? user.email;
```

```sql
-- Phase 2 — Backfill: Copy existing data
UPDATE users SET email_address = email WHERE email_address IS NULL;
```

```
# Phase 3 — Contract: Remove old column (after all code uses new)
prisma migrate dev --name remove_email_column

# schema.prisma
model User {
  id           String @id
  emailAddress String @unique  # new — now required
  # email removed
}
```

### Adding NOT NULL Constraint

```
# Phase 1 — Add nullable column
ALTER TABLE orders ADD COLUMN currency VARCHAR(3);

# Phase 2 — Backfill
UPDATE orders SET currency = 'USD' WHERE currency IS NULL;

# Phase 3 — Add constraint (after backfill complete)
ALTER TABLE orders ALTER COLUMN currency SET NOT NULL;
ALTER TABLE orders ALTER COLUMN currency SET DEFAULT 'USD';
```

## Concurrent Indexes

Always use `CONCURRENTLY` for indexes in production to avoid table locks:

```sql
-- ❌ BAD: Locks the table during index creation
CREATE INDEX idx_orders_tenant ON orders(tenant_id);

-- ✅ GOOD: Non-blocking index creation
CREATE INDEX CONCURRENTLY idx_orders_tenant ON orders(tenant_id);
CREATE UNIQUE INDEX CONCURRENTLY idx_users_email ON users(email);
```

In Prisma migrations, use raw SQL for concurrent index creation:

```typescript
// migration.sql (generated by prisma migrate dev)
-- CreateIndex (replace with concurrent version)
CREATE INDEX CONCURRENTLY "idx_orders_tenant_id" ON "orders"("tenant_id");
```

## Batch Data Migrations

Never migrate large datasets in a single transaction — it locks tables and risks timeouts:

```typescript
// ❌ BAD: Single transaction for large dataset
await prisma.$executeRaw`
  UPDATE users SET email_address = email WHERE email_address IS NULL
`;

// ✅ GOOD: Batch processing
async function backfillEmailAddress() {
  const BATCH_SIZE = 1000;
  let processed = 0;
  let hasMore = true;

  while (hasMore) {
    const batch = await prisma.user.findMany({
      where: { emailAddress: null },
      take: BATCH_SIZE,
      select: { id: true, email: true },
    });

    if (batch.length === 0) {
      hasMore = false;
      break;
    }

    await prisma.$transaction(
      batch.map((user) =>
        prisma.user.update({
          where: { id: user.id },
          data: { emailAddress: user.email },
        }),
      ),
    );

    processed += batch.length;
    console.log(`Backfilled ${processed} users`);

    // Small delay to reduce database pressure
    await new Promise((resolve) => setTimeout(resolve, 100));
  }

  console.log(`Backfill complete: ${processed} users updated`);
}
```

## Prisma Migration Workflow

### Development

```bash
# Create migration after schema change
npx prisma migrate dev --name descriptive_migration_name

# Apply pending migrations
npx prisma migrate deploy

# Reset database (dev only!)
npx prisma migrate reset

# Check migration status
npx prisma migrate status
```

### Production

```bash
# Always deploy migrations before deploying code
npx prisma migrate deploy

# Verify migration applied
npx prisma migrate status

# Never use migrate dev in production
# Never use migrate reset in production
```

### Migration Naming Conventions

```bash
# Format: <action>_<subject>
npx prisma migrate dev --name add_phone_to_users
npx prisma migrate dev --name remove_legacy_email_column
npx prisma migrate dev --name add_idx_orders_tenant_created
npx prisma migrate dev --name change_price_to_decimal
npx prisma migrate dev --name create_audit_logs_table
```

## Rollback Strategies

Prisma doesn't support automatic rollback. Plan ahead:

### Option 1: Forward-Only Migrations

Design all migrations to be forward-only (expand-contract ensures this):

- Never drop columns without a multi-phase plan
- Always add nullable columns first
- Keep old columns until new code is fully deployed

### Option 2: Manual Rollback Migration

```bash
# Create a rollback migration manually
npx prisma migrate dev --name rollback_add_phone_to_users
```

```sql
-- rollback_add_phone_to_users.sql
ALTER TABLE users DROP COLUMN IF EXISTS phone;
```

### Option 3: Database Snapshots

For major migrations, take a snapshot before deploying:

```bash
# Before migration
pg_dump $DATABASE_URL > backup_$(date +%Y%m%d_%H%M%S).sql

# After verifying success, clean up backup
# If rollback needed:
psql $DATABASE_URL < backup_20250115_103000.sql
```

## Schema Design Best Practices

### Primary Keys

```prisma
model User {
  id        String   @id @default(cuid())  // ✅ CUID for distributed systems
  // or
  id        String   @id @default(uuid())  // ✅ UUID v4
  // or
  id        Int      @id @autoincrement()  // ✅ Serial for simple cases
}
```

### Timestamps

```prisma
model User {
  id        String   @id @default(cuid())
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}
```

### Soft Deletes

```prisma
model User {
  id        String    @id @default(cuid())
  deletedAt DateTime? // null = active, non-null = soft deleted

  @@index([deletedAt]) // Index for filtering active records
}
```

```typescript
// Always filter soft-deleted records
const activeUsers = await prisma.user.findMany({
  where: { deletedAt: null },
});
```

### Multi-Tenancy

```prisma
model Category {
  id       String @id @default(cuid())
  tenantId String
  name     String

  @@unique([tenantId, name]) // Unique within tenant
  @@index([tenantId])        // Index for tenant queries
}
```

## Migration Checklist

Before running a migration in production:

- [ ] Migration tested in staging environment
- [ ] Migration is backward compatible (old code works with new schema)
- [ ] Large table migrations use batch processing
- [ ] New indexes use `CONCURRENTLY`
- [ ] Database backup taken before migration
- [ ] Rollback plan documented
- [ ] Migration deploy happens before code deploy
- [ ] Monitoring alerts set for migration duration
- [ ] Team notified of maintenance window (if needed)
